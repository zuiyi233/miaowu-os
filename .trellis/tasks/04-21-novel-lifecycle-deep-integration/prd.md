# 主项目写作流程与小说创作深度联动打通实施计划

## Goal

将 `deer-flow` 主写作链路与 `novel_migrated` 小说域能力彻底打通，实现“单一对话入口可完成小说全生命周期管理”，并保持与原版 `D:\deer-flow-main` 的核心协议兼容。

本计划采用“方案1（意图识别中间件）+ 方案2（Tool Calling 注册）”融合路线，不做二选一。

## Constraints

- 禁止在 WSL 内改动前端依赖（Win-only 依赖策略）。
- 本阶段目标为实施计划与任务编排，不做依赖安装。
- 允许静态验证，不宣称运行态“已完全验证”。
- 对已有用户改动保持兼容，不回滚无关变更。

## Current Snapshot (2026-04-21, Static Check)

- `/api/ai/chat` 已接入意图处理中间件调用链。
- 工具侧已出现 `create_novel` 等小说工具注册痕迹。
- 前端 `global-ai-service` 仍以通用聊天透传为主，结构化结果消费不足。
- 小说域仍存在 legacy 与 migrated 并行的契约收敛压力。

结论：项目已从“完全未接入”进入“部分接入但未闭环”状态，后续重点是补全全链路一致性和生命周期能力。

## Test-Issue Root Cause Mapping

|---|---|---|---|---|
| P1 | 主对话与小说模块自动化集成缺失 | 缺统一编排层，文本回复与业务执行脱节 | 方案1+方案2 | P0 |
| P2 | 无稳定意图识别与会话状态管理 | 请求链路缺状态机和缺字段追问门禁 | 方案1 | P0 |

## Target Architecture

- 入口层：统一聊天入口（主项目工作区 + 小说工作台）。
- 编排层：Intent Session Engine（方案1）与 Tool Dispatcher（方案2）协同。
- 领域层：Novel Domain Orchestrator 统一调用 `novel_migrated`。
- 数据层：业务库 + 事件账本（幂等、审计、补偿）。

核心决策：

- `Project` 作为小说域 canonical root。
- 写操作遵循“确认前不落库、确认后单次落库、重试不重复写”。
- 技能加载遵循主项目设置中心开关，不做代码级强制全开。

## Phase Plan (Trellis)

### Phase 0: 基线冻结与契约盘点（1-2天）

交付项：

- 双链路现状图（`/api/threads/*/runs/stream` vs `/api/ai/chat`）。
- 跨层字段字典（thread/project/novel/chapter/session/tool_calls）。
- P0/P1 风险清单与 owner。
- 灰度开关与回滚入口清单。

验收标准：

- 关键边界“前端 -> 网关 -> 中间件 -> 小说域 API -> DB”可追踪。
- 每个高风险点有明确责任人和截止时间。

风险：

- 文档与代码漂移导致错误基线。
- legacy 路由掩盖真实问题。

回滚策略：

- 本阶段只读，不改逻辑。
- 基线文档锁版，后续按差异审查。

依赖前置：

- 网关路由清单。
- 关键 API 请求/响应样例。
- 最小复现实例数据。

### Phase 1: 协议统一与 P0 契约修复（3-5天）

交付项：

- `DomainAction` 与 `DomainToolCall` 统一协议。
- `/api/ai/chat` 与小说域响应 envelope 对齐。
- 角色/关系/组织等高风险契约修复与迁移脚本。
- `tool_calls` 统一为 Deer-Flow 兼容结构 `{name, args, id}`。

验收标准：

- P0 契约错位接口清零。
- 前后端类型定义可自动校验。
- 兼容 alias 可观测、可追踪。

风险：

- 兼容旧前端时出现回归。
- DDL 与历史脏数据冲突。

回滚策略：

- 新契约挂 feature flag（如 `vNext`）。
- 保留旧字段读取路径过渡。
- 迁移脚本可逆执行。

依赖前置：

- Phase 0 风险清单。
- 调用频次与错误分布统计。
- 历史数据抽样结果。

### Phase 2: 会话状态机与写入门禁（4-7天）

交付项：

- `normal/create/manage` 三态状态机。
- 缺字段追问机制（书名/类型/题材/受众/字数目标等）。
- 确认门禁、幂等键、防重复写。
- 会话持久化与超时回收策略。

验收标准：

- 字段不全时稳定追问。
- 未确认不落库。
- 确认后单次写入且可审计。
- 重启后会话恢复策略明确。

风险：

- 状态机与现有中间件逻辑冲突。
- 并发导致错会话写入。

回滚策略：

- 保留直通降级路径。
- 按租户/用户灰度启用状态机。
- 异常时回退 `normal` 模式。

依赖前置：

- Phase 1 协议稳定。
- session key 优先级策略定稿（thread/context/hash fallback）。

### Phase 3: 前端联动与结构化消费（3-5天）

交付项：

- `global-ai-service` 结构化响应消费能力升级。
- 主聊天与小说工作台的会话状态提示统一。
- 上下文透传贯通（project/novel/chapter/thread）。
- 错误与降级反馈 UI 标准化。

验收标准：

- 主聊天与小说工作台看到一致结构化反馈。
- 上下文切换不串会话。
- Tool 调用结果可见、可追踪。

风险：

- 仅文本渲染逻辑回归。
- SSE 事件兼容问题。

回滚策略：

- 前端保留 `legacy_text_mode`。
- 后端可回退到仅 `content` 响应。
- 页面级灰度发布。

依赖前置：

- Phase 2 会话契约稳定。
- 前端消息渲染组件评估完成。

### Phase 4: 生命周期闭环（5-8天）

交付项：

- 项目级 JSON 导入/导出/校验。
- 长任务状态机（提交/执行/失败补偿/恢复）。
- 数据一致性巡检与修复作业（参考 MuMuAINovel 成熟方案）。

验收标准：

- 导入导出可回环。
- 长任务可恢复。
- 关系/组织一致性可自动巡检并产出报告。

风险：

- 历史数据质量不足导致导入失败。
- 补偿逻辑覆盖不全。

回滚策略：

- 导入默认 dry-run。
- 写入前快照备份。
- 失败按任务粒度回滚与重放。

依赖前置：

- Phase 1 契约统一。
- Phase 3 上下文贯通。
- 数据备份机制可用。

### Phase 5: 可观测、灰度与发布收口（2-4天）

交付项：

- 全链路日志维度（request_id/thread_id/project_id/session_key/idempotency_key）。
- 指标面板（成功率、重试率、重复写拦截率、P95）。
- 灰度策略与 15 分钟回退 Runbook。

验收标准：

- 支持租户/用户粒度灰度。
- 异常可快速定位到边界层。
- 回退演练通过且无数据损坏。

风险：

- 指标定义不一致导致误判。
- 告警噪音影响发布节奏。

回滚策略：

- 一键关停联动能力，保留原主链路。
- 启用只读保护并暂停高风险写操作。
- 事件账本支持事后补偿。

依赖前置：

- Phase 0-4 埋点落地。
- SLO/SLA 阈值设定完成。

## Parallel Workstreams (Max 4)

1. 工作流A（协议与契约修复）：后端核心协议、API-Model 对齐、迁移脚本与契约测试。
2. 工作流B（会话编排与门禁）：状态机、确认执行、幂等防重、会话恢复。
3. 工作流C（前端联动）：结构化响应消费、SSE 解析升级、状态 UI。
4. 工作流D（平台保障）：导入导出、任务补偿、观测与灰度发布。

## Acceptance Criteria

- 用户在主对话发起“创建小说/改章节/改大纲/改角色/改关系/改组织/改物品”时可触发真实业务动作，而非仅文本建议。
- 会话态可持续追问缺失字段，确认前不落库，确认后单次落库。
- 技能调用遵循主项目开关配置，不绕过设置中心。
- 新增联动能力可灰度、可监控、可回滚，且主项目原有对话能力不受破坏。

## Definition of Done

- 计划范围内的 Phase 交付物全部落地并通过对应验收。
- 关键跨层契约有自动化校验或可复现手工验证证据。
- 发布前完成灰度与回滚演练记录。
- 未解决的风险、限制与验证缺口均文档化。

## Validation Strategy

- 静态验证：类型/契约/代码审查、关键路径 py_compile。
- 运行态验证：在 Windows 主环境执行，不在 WSL 调整前端依赖。
- 最小联调矩阵：`/api/ai/chat`、关键小说 CRUD、E2E 对话创建小说链路。
