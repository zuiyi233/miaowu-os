# 阶段二：小说创作质量闭环与生产化联动

## Goal

在“首期联动打通已完成”的基础上，进入第二阶段交付：

1. 自动生成-审核-修订闭环（减少手工串联）
2. 跨章一致性检查（角色/物品/时间线）
3. 定稿门禁（可阻断、可解释）
4. 长任务恢复与失败补偿（可追踪、可恢复）
5. 可观测与灰度回滚（可发布、可运营）

最终目标：将能力从“可用”提升到“稳定可规模化使用”。

## Scope

### In Scope

1. 后端：流水线编排、状态机、跨章一致性引擎、定稿检查、任务恢复、观测埋点
2. 前端：最小联动改造（任务状态展示、阻断提示、报告入口）
3. 协议：统一任务状态与报告结构，保证主项目聊天与小说页消费一致

### Out of Scope

1. 不重做全量小说业务 UI
2. 不新增前端依赖，不改 Win-only 前端依赖模式
3. 不一次性重构全部 legacy 路由，仅在必要链路做可控收敛

## Constraints

1. 环境约束：禁止在 WSL 中变更前端依赖
2. 兼容约束：保持 Deer-Flow 主链路兼容，不做破坏式替换
3. 发布约束：所有高风险能力必须挂 feature flag
4. 数据约束：写操作需幂等，支持失败补偿与审计追踪

## Deliverables

### WP1 自动生成-审核-修订流水线（P0）

交付项：

1. 章节生成后自动触发分析任务
2. 分析结果自动生成可执行修订建议
3. 用户确认后执行修订并更新任务状态

验收：

1. 单章操作可在一次流程中完成“生成 -> 分析 -> 建议 -> 修订”
2. 失败状态可重试，且无重复写入

### WP2 跨章一致性检查引擎（P0）

交付项：

1. 角色状态快照（按章）
2. 物品状态快照（按章）
3. 事件时间线快照（按章）
4. 项目级一致性报告 API

验收：

1. 能识别至少三类冲突：角色设定冲突、物品状态冲突、时间线冲突
2. 报告返回冲突定位（章节/实体/字段）与修复建议

### WP3 定稿检查清单门禁（P0）

交付项：

1. 定稿前自动检查：一致性、伏笔回收、低分章节、敏感词、格式规则
2. 阻断项标准化返回结构
3. “通过/阻断/警告”三级结果

验收：

1. 阻断项存在时不允许定稿完成
2. 返回可解释错误信息并可直接定位处理入口

### WP4 长任务恢复与补偿（P0）

交付项：

1. 批量生成/批量分析任务状态机
2. 断点续跑（从失败节点恢复）
3. 幂等键与资源级防重保护

验收：

1. 中断后可继续执行，不重复写历史成功章节
2. 任务重试后状态与结果一致

### WP5 可观测与灰度发布（P1）

交付项：

1. 统一链路日志字段：`request_id/thread_id/project_id/session_key/idempotency_key`
2. 指标：成功率、失败率、重试率、P95、重复写拦截率
3. 灰度开关与回滚 Runbook

验收：

1. 可按用户粒度灰度
2. 15 分钟内可完成回滚

### WP6 前端最小联动（P1）

交付项：

1. 任务状态条与关键错误提示
2. 一致性报告入口（聊天页/小说页）
3. 定稿门禁阻断信息展示

验收：

1. 不新增前端依赖
2. 不改前端依赖安装模式
3. 主聊天和小说工作台均可看到结构化状态

## Risks & Mitigations

1. 风险：跨章一致性误报偏高  
   对策：阈值可配置 + 规则白名单 + 报告分级（阻断/警告）
2. 风险：长任务恢复造成脏状态  
   对策：事件日志 + 幂等键 + 恢复前快照校验
3. 风险：前端显示状态与后端不一致  
   对策：统一状态枚举与错误码，增加契约测试
4. 风险：灰度期间回滚不彻底  
   对策：写路径受 feature flag 控制，预置一键降级脚本

## Definition of Done

1. WP1-WP4 核心功能可用并有自动化测试覆盖
2. WP5 灰度与回滚演练通过
3. WP6 在不新增依赖前提下完成最小联动
4. 关键指标可观测，且故障可定位到模块边界
5. 更新 `.trellis/spec/` 对应条目并记录会话
