- 以 `小说功能对比分析文档.md` 的“5.1 缺失能力清单”为唯一迁移输入范围。

## 5. 分波次实施（Diverge -> Converge 后的 MVP 收敛）

## Wave 1（MVP，先打核心小说生产链）

- 职业体系
- 伏笔状态机
- 记忆分析与向量检索
- MCP 工具加载链
- （已确认）不包含 `auth/users/admin`

### 目标

- 先完成“写作核心链路闭环”：章节分析 -> 记忆写入/检索 -> 伏笔管理 -> AI 调用工具链。

## Wave 2（生产增强）

- 拆书导入
- 封面生成
- 灵感生成

### 目标

- 补齐创作前与创作中增强能力，不改主链结构。

## Wave 3（账户域）

- `auth/users/admin` 与用户模型链

### 目标

- 在确认主项目账号体系边界后再并入，避免提前引入冲突。

## 6. 不重复造轮子清单（必须遵守）

- 不新建第二套 `models` 列表与模型配置路由。
- 不新建第二套 MCP 配置管理接口。
- 不新建与主项目 `memory` 基础能力平行的全局 memory 管理接口。
- 不重写参考项目业务服务算法（职业生成、伏笔状态流转、记忆分析逻辑）；仅做接入适配。

## 7. 关键风险与验证缺口

### 7.1 架构差异风险

- 参考项目依赖 `app.database`、SQLAlchemy Base、完整 models/schemas/services 栈。
- 主项目当前仅显式存在 `gateway` 与 `channels` 目录，缺同层基础模块。
- 结论：可 1:1 复制源码，但必须补最小基础支撑（DB Base、依赖注入绑定、路由注册）。

### 7.2 依赖风险

- `memory_service.py` 依赖 `chromadb` 与 `sentence_transformers`。
- 相关 provider/邮件/oauth 等依赖是否已在主项目后端依赖清单中，需要实施前核验。

### 7.3 验证缺口

- 本文基于静态核验，未做运行联调。
- 未验证迁移后接口兼容性与前端调用一致性。
- 未验证数据迁移（旧数据与新表结构）策略。
