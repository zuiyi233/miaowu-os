# 混合嵌入系统实现总结

## 概述

我们成功实现了一个基于"本地向量检索 + 关键词匹配"的混合检索（Hybrid Search）系统，并引入了"动态状态切片"机制来解决上下文污染和状态滞后问题。

## 核心架构

### 1. 数据层改造 ✅

**文件**: `lib/schemas.ts`

- 为所有实体（角色、场景、势力、物品）添加了 `embedding` 和 `lastEmbedded` 字段
- 使用 `embeddingMixin` 混入对象确保代码复用

**文件**: `lib/storage/db.ts`

- 升级数据库版本到 v13，支持新的 embedding 字段
- Dexie 自动处理新字段的添加，现有记录的 embedding 字段将为 undefined

### 2. 向量计算工具 ✅

**文件**: `lib/utils/math.ts`

- 已存在完整的余弦相似度计算函数 `cosineSimilarity`
- 支持欧几里得距离、曼哈顿距离等多种向量计算方法
- 包含向量归一化功能

### 3. 增强 Embedding 服务 ✅

**文件**: `services/embeddingService.ts`

- **批量生成**: `syncAllEmbeddings()` 方法支持为所有实体批量生成向量
- **缓存逻辑**: 通过 `lastEmbedded` 时间戳判断是否需要更新向量
- **混合检索**: `semanticSearch()` 方法结合关键词匹配和向量检索
- **智能文本构建**: `buildEmbeddingText()` 为不同类型实体构建语义丰富的文本

### 4. ContextEngine 重构 ✅

**文件**: `services/contextEngineService.ts`

- **混合检索**: `semanticRetrieve()` 方法结合本地匹配和向量检索
- **动态状态注入**: `assembleDynamicContext()` 方法解决上下文污染问题
- **上下文感知重排序**: 根据当前章节出现频率和实体重要性进行重排序
- **时间线集成**: 利用时间线事件获取实体最新状态

### 5. 用户体验优化 ✅

**脏数据队列机制**:

- **文件**: `stores/useEmbeddingStore.ts`
- **功能**: 管理需要重新生成向量的实体队列
- **特性**: 支持去重、重试机制、队列状态管理

**后台处理器**:

- **文件**: `components/common/BackgroundEmbedder.tsx`
- **功能**: 利用 `requestIdleCallback` 在浏览器空闲时处理向量化
- **特性**: 不阻塞 UI，有重试机制，自动错误处理

**混合检索 Hook**:

- **文件**: `hooks/useHybridSearch.ts`
- **功能**: 提供乐观 UI，先显示关键词匹配结果，后补充向量检索结果
- **特性**: 2 秒防抖，自动合并和去重结果

### 6. 表单集成 ✅

**文件**: `components/CharacterEditForm.tsx`

- **集成**: 保存成功后自动将实体推入脏数据队列
- **优化**: 构建语义丰富的文本用于向量化
- **用户体验**: 保存操作零延迟，向量化后台处理

**文件**: `App.tsx`

- **集成**: 在应用顶层添加 `<BackgroundEmbedder />` 组件
- **功能**: 确保后台向量化处理器在整个应用生命周期中运行

## 核心特性

### 1. 混合检索 (Hybrid Search)

- **Stage 1 (0ms)**: 关键词匹配，立即返回结果
- **Stage 2 (Debounce)**: 向量检索，2 秒防抖后执行
- **重排序**: 结合上下文感知和实体重要性

### 2. 动态状态注入

- **时间线集成**: 从时间线事件中获取实体最新状态
- **精简信息**: 避免长篇背景故事的上下文污染
- **状态感知**: 注入"当前状态"而非静态设定

### 3. 成本与性能优化

- **写入成本**: 脏数据队列 + 闲时处理，避免阻塞用户操作
- **查询延迟**: 乐观 UI + 防抖，平衡速度与准确性
- **API 调用**: 智能缓存，避免重复向量化

## 技术亮点

### 1. 零重型依赖

- 完全基于现有的 Zustand + Dexie + React Hooks
- 无需引入额外的向量数据库或重型库
- 向量计算在本地完成，仅 API 调用用于生成向量

### 2. 渐进式增强

- 保持向后兼容，现有功能不受影响
- 新功能作为增强层，可选择性启用
- 优雅降级，API 不可用时回退到关键词匹配

### 3. 用户体验优先

- 编辑操作零延迟
- 检索结果即时显示
- 后台处理不干扰用户操作

## 解决的核心问题

### 1. 语义丢失 ✅

- **问题**: 传统 RAG 无法理解"拿着锤子的光头"指向"王铁"
- **解决**: 向量检索理解语义相似性，不依赖精确匹配

### 2. 上下文污染 ✅

- **问题**: 长篇背景故事淹没关键信息
- **解决**: 动态状态注入，只提供核心特征和当前状态

### 3. 状态滞后 ✅

- **问题**: AI 不知道角色的实时状态变化
- **解决**: 时间线集成，提供"此时他正在..."的动态信息

### 4. 成本控制 ✅

- **问题**: 频繁 API 调用导致费用高昂
- **解决**: 脏数据队列 + 闲时处理 + 智能缓存

### 5. 查询延迟 ✅

- **问题**: 向量检索 API 调用导致响应缓慢
- **解决**: 乐观 UI + 混合检索，先快后准
