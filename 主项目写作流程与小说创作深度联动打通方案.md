# 主项目写作流程与小说创作深度联动打通方案

- 文档版本: v1.0
- 编写日期: 2026-04-21
- 适用范围: `D:\miaowu-os`（主项目 deer-flow 二开）
- 目标: 实现“主项目写作流程 + 小说创作全生命周期”在会话、工具、数据、权限、观测、发布上的彻底打通

---

## 1. 背景与目标

当前系统已经具备两套能力：

1. 主项目线程式写作链路（`/api/threads/*/runs/stream`，agent + tools + middleware）
2. 小说创作链路（`/api/ai/chat` + `novel_migrated` 业务 API）

两套能力都能工作，但仍是并行体系，导致：

1. 用户在不同页面/入口触发不同链路，体验割裂
2. 结构化结果（tool_calls/novel/session）无法完整透传到所有前端链路
3. 小说领域部分 API 存在“路由存在但数据契约不一致”的高风险点
4. 项目级导入导出与迁移闭环不完整

### 1.1 目标（Must Have）

1. 用户通过单一对话入口即可完成小说全生命周期操作（项目/大纲/章节/角色/关系/组织/物品等）
2. 既支持“意图会话引导（方案1）”，也支持“Tool Calling 自主调用（方案2）”，并可协同
3. 技能加载默认按主项目开关策略生效，不绕过设置体系
4. 保持与现有 Deer-Flow 主链路兼容，支持灰度与快速回滚
5. 提供可观测、可审计、可恢复的工程闭环

### 1.2 非目标（Not in Scope, v1）

1. 不在首期重做全部小说业务 UI
2. 不在首期引入新依赖或变更前端依赖安装模式
3. 不在首期一次性重构全部 legacy 路由，仅做可控收敛

---

## 2. 现状诊断与根因

## 2.1 关键现状

1. 主线程链路和小说链路并行存在，不是同一状态机
2. `IntentRecognitionMiddleware` 已具备创建/管理会话能力，但前端消费结构化信息仍不完整
3. 小说域以 `novel_migrated` 为现代主链路，同时仍保留 `routers/novel.py` 的历史存储兼容路径
4. 局部领域对象存在 API 与 ORM 字段错位风险（尤其角色/关系/组织）

## 2.2 根因分类

1. 架构根因: 历史演进形成双入口双协议，缺少“统一领域编排层”
2. 契约根因: 前后端/路由/模型之间缺统一版本化契约
3. 协议根因: 聊天文本流与结构化动作流没有统一消费语义
4. 生命周期根因: 导入导出、批量分析、任务状态等闭环未统一规划

---

## 3. 总体架构（目标态）

采用“一个领域根 + 两种交互机制 + 一个统一协议层”的架构。

### 3.1 核心原则

1. `Project` 作为小说域唯一 canonical root
2. `novel_migrated` 作为小说业务主实现
3. `routers/novel.py` 降级为兼容层，逐步收敛
4. 意图中间件与 Tool Calling 并存，不二选一
5. 所有写操作统一“确认执行 + 幂等保护 + 审计事件”

### 3.2 目标拓扑

```text
[前端统一聊天入口 / Novel工作台]
          |
          v
  [Unified Chat Gateway]
          |
   +------+-------------------+
   |                          |
   v                          v
[Intent Session Engine]   [Tool Calling Dispatcher]
   |                          |
   +-----------+--------------+
               v
     [Novel Domain Orchestrator]
               |
       [novel_migrated APIs]
               |
       [DB + Event Ledger]
```

### 3.3 三态会话模型

1. `normal`: 普通对话
2. `create`: 小说创建会话（字段收集 -> 确认 -> 落库）
3. `manage`: 生命周期管理会话（动作识别 -> 缺字段追问 -> 确认 -> 落库）

会话状态必须包含：

1. `thread_id` / `session_key`
2. `user_id` / `project_id` / `novel_id`
3. `pending_action` / `missing_fields` / `awaiting_confirm`
4. `idempotency_key` / `last_event_id`

## 4. 方案1 + 方案2 融合设计

### 4.1 方案1（意图识别中间件）定位

职责：

1. 识别“创建/管理”意图并进入会话状态
2. 对缺失字段做逐步追问
3. 在写操作前执行确认门禁
4. 在不命中意图时完全透传主链路

适用场景：

1. 用户自然语言描述不完整
2. 需要高确定性字段收集
3. 需要业务级 guardrail（确认、上下文绑定、权限预校验）

### 4.2 方案2（Tool Calling 注册）定位

职责：

1. 注册标准化领域工具（如 `project.update`、`chapter.create`）
2. 让模型在上下文充分时自主调用动作
3. 提供结构化执行结果与错误类型

适用场景：

1. 指令明确、参数完整
2. 需要 agent 自主编排多步骤
3. 需要跨工具联动（任务拆分、文件产物、图像等）

### 4.3 融合决策规则

1. 中间件优先判断是否进入 `create/manage` 会话
2. 若处于会话态，优先走会话状态机
3. 若不在会话态且请求明确，允许 Tool Calling 直接执行
4. 会话内也允许调用工具，但必须经过“缺字段检查 + 确认门禁”

## 5. 统一协议与数据契约

## 5.1 统一动作协议（DomainAction）

建议标准字段：

1. `action`: 例如 `project.update`, `chapter.create`
2. `entity`: `project/chapter/outline/character/relationship/organization/item`
3. `operation`: `create/update/delete/list/generate/...`
4. `scope`: `{user_id, project_id, thread_id, novel_id}`
5. `payload`: 参数对象
6. `idempotency_key`: 幂等键
7. `requires_confirmation`: 是否必须确认

## 5.2 统一工具调用协议（DomainToolCall）

对外统一为 DeerFlow 兼容结构：

```json
{
  "name": "chapter.create",
  "args": {
    "project_id": "...",
    "title": "..."
  },
  "id": "call_xxx"
}
```

扩展结果单独字段承载，不污染 `tool_calls` 基础结构。

## 5.3 统一上下文传递

必须贯通：

1. 前端 `context`
2. `/api/ai/chat` 请求体
3. `/api/threads/*/runs/*` 配置上下文
4. 中间件 session key
5. Domain Orchestrator 执行上下文

建议允许字段白名单扩展：

1. `novel_id/novelId`
2. `project_id/projectId`
3. `chapter_id/chapterId`
4. `writing_mode`
5. `scene_id`（可选）

## 6. 生命周期能力清单（目标闭环）

## 6.1 实体能力矩阵（v1 必达）

1. 项目: 创建/更新/删除/查询/切换/统计
2. 大纲: 创建/更新/删除/排序/展开
3. 章节: 创建/更新/删除/局部重写/批量生成/状态管理
4. 角色: 创建/更新/删除/生成功能
5. 关系: 创建/更新/删除/图谱查询
6. 组织: 创建/更新/删除/成员增删改
7. 物品: 通过 foreshadow(category=item) 完成增删改查

## 6.2 支撑能力（v1.5/v2）

1. 项目级 JSON 导入/导出/校验
2. 角色级导入/导出/校验
3. 批量分析、任务历史、标注、导航闭环
4. 图谱布局持久化
5. 事件回放与审计导出

## 7. 关键工程问题与解决策略

| 问题 | 根因 | 解决策略 | 对应方案 |
|---|---|---|---|
| 主写作链路与小说链路割裂 | 双入口双状态机 | 引入统一编排层与统一上下文模型 | 方案1+2 |
| 结构化结果前端丢失 | 文本流消费逻辑单一 | 前端增加结构化分支消费与会话状态渲染 | 方案1+2 |
| API 与模型字段错位 | 历史演进未收敛 | 做契约校对与迁移脚本，先修 P0 错位 | 两者共用基础 |
| 技能加载绕过风险 | 多处加载策略不一致 | 统一按 `extensions_config` 开关，失败走 fail-open | 方案1 |
| 重试导致重复写入 | 缺幂等策略 | 引入 `idempotency_key` + 资源级锁 | 两者共用基础 |
| 灰度风险高 | 缺回滚路径 | Feature flag + shadow 校验 + 一键回退 | 两者共用基础 |

## 8. 分阶段实施计划（可执行）

## Phase 0: 基线冻结（1-2 天）

交付：

1. 现状链路图
2. API/模型契约差异表
3. 风险分级列表（P0/P1/P2）

验收：

1. 明确 canonical 与 legacy 边界
2. 明确可灰度切换点

## Phase 1: 契约修复与统一协议（3-5 天）

交付：

1. DomainAction / DomainToolCall 规范
2. 角色/关系/组织 P0 契约修复
3. `tool_calls` 线协议统一

验收：

1. 关键实体 CRUD 静态契约一致
2. 无破坏性兼容回归

## Phase 2: 会话状态机与工具协同（4-7 天）

交付：

1. create/manage/normal 三态完整状态机
2. 缺字段追问与确认门禁全覆盖
3. 方案1 与方案2 协同决策器

验收：

1. 管理动作在字段不完整时可稳定追问
2. 确认前不落库，确认后单次落库

## Phase 3: 上下文贯通与前端联动（3-5 天）

交付：

1. 上下文字段贯通（thread/project/novel/chapter）
2. 前端结构化响应消费
3. 会话状态 UI 提示与错误提示

验收：

1. 在主聊天与小说工作台均可看到一致的结构化反馈
2. project/novel 上下文切换不串会话

## Phase 4: 导入导出与任务闭环（5-8 天）

交付：

1. 项目级 JSON 导入/导出/校验 API
2. 批量分析/任务状态/历史查询统一接口
3. 失败补偿策略

验收：

1. 项目可迁移、可恢复
2. 长任务可追踪、可恢复

## Phase 5: 可观测与灰度发布（2-4 天）

交付：

1. 端到端链路日志（request_id / thread_id / project_id）
2. 指标面板（成功率、重试率、重复写拦截率、动作耗时）
3. 灰度开关与回滚手册

验收：

1. 可按租户/用户灰度
2. 15 分钟内可安全回退

---

## 9. 兼容策略与回滚策略

## 9.1 兼容策略

1. 保留 legacy 路由，但仅作为兼容层
2. 统一对外动作协议，不强迫前端一次性重写
3. 对旧字段做 alias 兼容（限过渡期）

## 9.2 回滚策略

1. 所有新链路受 feature flag 控制
2. 支持 shadow read/write 对比
3. 出现异常时切回旧链路
4. 依赖事件日志做补偿与重放

---

## 10. 验收标准（Definition of Done）

## 10.1 功能验收

1. 用户通过单一对话可完成完整小说生命周期管理
2. create/manage 会话追问逻辑稳定，确认门禁生效
3. tool calling 与会话模式可协同，不互相覆盖

## 10.2 质量验收

1. 无新增高危字段错位
2. 幂等键命中时无重复写入
3. 错误信息分层清晰（用户友好 + 调试可定位）

## 10.3 运行验收

1. 关键动作成功率达到目标阈值（建议 >= 99%）
2. 关键动作 P95 延迟可控
3. 灰度与回滚演练通过

---

## 11. 并行开发建议（执行层）

可按 4 条并行工作流推进：

1. 协议与中间件组: 状态机、动作协议、确认门禁、幂等
2. 小说领域组: 项目/章节/角色/关系/组织/物品 API 契约修复
3. 前端联动组: 结构化响应消费、会话状态 UI、上下文贯通
4. 平台保障组: 观测、灰度、回滚、导入导出工具链

每组以“可回退小步提交”推进，周级合并里程碑。

---

## 12. 结论

推荐实施路径：

1. 先稳契约（P0），再做融合（方案1+2 协同），最后补闭环（导入导出/分析任务/观测）
2. 保持对主项目线程式写作链路的兼容，不做破坏性替换
3. 以 `Project` 为 canonical root，推动小说域逐步完成真正的一体化生命周期平台化
